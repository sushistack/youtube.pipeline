### Second code review pass (2026-04-24 — Blind Hunter / Edge Case Hunter / Acceptance Auditor)

- **Concurrent `pipeline export` invocations race on output file** — `os.WriteFile` (JSON) and `os.Create` + `csv.WriteAll` (CSV) are non-atomic; overlapping invocations produce truncated/garbled output mid-write. Fix is write-to-temp + `os.Rename` for atomic publish. Single-operator use makes this very low risk; defer until multi-operator export or long-running export becomes a pattern ([internal/service/export_service.go:384-418](../../internal/service/export_service.go)).
- **`created_at` TEXT lexicographic ordering not chronological across mixed formats** — `ListByRunIDForExport ORDER BY created_at ASC, id ASC` is byte-stable within an invocation; chronological correctness holds only if every row stores the same ISO-8601 UTC-with-`Z` format. SQLite `CURRENT_TIMESTAMP` (no `Z`), app-set RFC3339 (`Z`), and `datetime('now')` (space separator) would sort incorrectly if mixed. Pre-existing TEXT storage convention; address when a migration normalizes timestamp columns ([internal/db/decision_store.go:139-141](../../internal/db/decision_store.go)).
- **No fault-injection coverage for `writeCSVFile` Close error path** — The named-return close-error capture is correct by inspection; exercising it requires a mock `*os.File` / filesystem fault injector not currently in the test kit. Low priority ([internal/service/export_service.go:399-418](../../internal/service/export_service.go)).
- **Unicode runID normalization gap (macOS HFS+/APFS)** — `isSafeRunID` accepts any Unicode code point; two runIDs differing only in NFC/NFD or narrow-nbsp-vs-ASCII-dash can collide at the filesystem layer on case-insensitive/normalizing volumes. Defender-depth concern only — SCP IDs are ASCII in practice ([internal/service/export_service.go:262-279](../../internal/service/export_service.go)).
